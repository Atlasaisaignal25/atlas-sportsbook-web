# Atlas AI MLB Engine Audit

Date: 2026-07-10

## Scope

This audit covers the current Atlas MLB signal path before SportsDataIO betting-data expansion. The goal was to verify actual data flow, remove misleading assumptions, and calibrate only with inputs already available in production.

## Current Data Flow

1. Odds ingestion runs through `app/api/cron/automationUtils.ts`.
2. MLB odds are fetched from The Odds API using `baseball_mlb` with `h2h`, `spreads`, and `totals`.
3. Public MLB rows are written to `mlb_public_signals`.
4. Daily Top 5 reads `mlb_public_signals`, dedupes by game, sorts by Atlas score, and writes `mlb_top5_live`.
5. Top Signal and Top Play remain separate precision products. Their precision engine reads the candidate pool and public signal sources, but the subscription Top 3/Top 5 path does not unlock those products.
6. Pregame validation checks live odds within 30 minutes of start and marks rows `CONFIRMED`, `DOWNGRADED`, or `REMOVED`.
7. History grading uses completed scores from SportsDataIO when available, with The Odds API scores as fallback.
8. Market Impact odds movement is stored in `public.market_odds_snapshots` and converted into consensus movement events.

## Verified MLB Markets

Active automated MLB markets:

- Moneyline: `h2h`
- Run line: The Odds API `spreads`
- Full-game totals: `totals`

Not currently generated by the automated MLB engine:

- First 5 innings
- Team totals
- Player props
- Live betting
- Pitcher props
- Alternate markets

## Important Findings

- The previous MLB ranking relied mostly on American odds implied probability, market weight, price safety, and price range.
- No confirmed starting pitcher, lineup, weather, injury, bullpen, travel, umpire, or team rolling-stat data was available inside the automated MLB signal path.
- Candidate Pool types contained fields such as `steamScore`, `sharpScore`, `rlmScore`, `liquidityScore`, `closingLineProjection`, and `marketConsensus`, but those values were not populated with verified production inputs.
- Market Impact odds movement existed as a separate pipeline, but it was not feeding the MLB signal generator.
- The engine already abstains naturally when fewer than five qualified candidates exist; it does not need to force five picks.

## Calibration Applied

The MLB generator now uses:

- American odds to implied probability through a shared helper.
- Two-way no-vig consensus probabilities when the available market has a valid two-sided pair.
- Median and average price context across books.
- Book coverage and price spread.
- MLB minimum coverage of two books.
- Stale odds rejection when Odds API timestamps show a market older than 240 minutes.
- Real recent odds movement context from Supabase snapshots when available.

Movement is bounded context only. Single-book low-impact movement does not add score. Missing movement does not penalize a candidate or create fake movement.

## Leakage and False Intelligence Controls

The current engine does not use unavailable MLB inputs as scoring facts. Documentation and audit output explicitly mark these unavailable:

- confirmed starting pitchers
- official lineups
- injuries
- weather
- team/player rolling statistics
- player props
- live odds

The automated copy now frames movement and no-vig consensus as market context, not guaranteed edge.

Authorization:

The route returns:

- verified data sources
- unavailable inputs not used
- active selection rules
- Supabase snapshot status
- recent consensus movements
- recent public MLB signals
- recent live Top 5 rows

No API keys are returned.

## Remaining Pre-SportsDataIO Limitations

- MLB predictive edge remains market-derived until verified pitcher, lineup, weather, roster, and team-stat data are connected.
- Run line and totals are supported only from full-game Odds API markets.
- News/Market Impact events are intentionally not used as direct pick boosts unless they can be matched to verified market movement in future work.
- Top Signal and Top Play precision scoring still depends on candidate-pool/public-signal inputs and should be recalibrated again after SportsDataIO betting context is connected.
